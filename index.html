Quick Start

Start here. Don’t overthink it. This section gives you the default structure, the permission model, and the most common mistakes to avoid.

What this system does

Roles control three things:

  • What users can access
  • What users can do
  • How cases move through the system

If you get roles wrong, people either can’t do their job or they end up with too much control.

The only thing that matters: Cases

Your role design should answer three questions:

  • Who creates cases?
  • Who works cases?
  • Who manages and escalates cases?

If you don’t answer those clearly, the rest falls apart.

How permissions work

Keep the model simple: Workspace → Resource → Action. Example: Main Workspace → Cases → Escalate.

Permissions UI overview

Pick a workspace, toggle a resource on, then decide how much control it gets.

You have three permission levels to work with:

  • Full Control — fast, but risky
  • Custom — recommended when workflow matters
  • Read Only — visibility without action
Biggest mistakes to avoid: giving everyone Full Control, over-customizing on day one, and building roles that do not match real job responsibilities.

Role Reference

These four roles are enough for most organizations. Keep them distinct. The more you blur them together, the harder it is to control case workflow.

Admin

Full system ownership

Admin controls users, roles, templates, and system settings. This role has access across all workspaces and can create, update, assign, escalate, and configure nearly everything.

  • Use for platform owners only
  • Keep the user count extremely small
  • Best treated as emergency-level access
Reality: this is your break-glass role. Keep it extremely limited.

Admin role view with broad access across the main workspace.
Detective

Execution

Detective is the role for assigned case work. It should be the largest user group and it should stay simple. This role works cases, updates status, and responds to follow-up.

  • Keep permissions narrow and obvious
  • Avoid granting escalation authority
  • Design this role for speed and clarity
Reality: this should be your largest user group with the simplest permissions.

Lab Admin

Workflow control

Lab Admin is your operational role. This is the group that manages the case lifecycle, assigns detectives, handles escalations, and pushes follow-up through the system.

  • Ideal for operations leads
  • Should own workflow movement, not full system control
  • Usually the most important role in day-to-day use
Reality: this is your most important operational role.

Manager

Oversight and visibility

Manager is for dashboards, reports, and leadership visibility. This role may have limited case interaction, but it should not automatically receive operational permissions.

  • Use for leadership and supervisors who need visibility
  • Keep operational permissions limited unless there is a real need
  • Default toward visibility, not action
Reality: avoid giving managers operational permissions unless necessary.

Case Permissions

This is the most important detail in the whole guide. Cases support the deepest level of control, and these permissions define your actual workflow behavior.

Create / Update / Delete

Controls who can open cases, edit them, and remove them. Use carefully.

Archive / Restore

Controls who can move cases out of active view and return them later.

Assign

Controls who can attach detectives to cases and move work into execution.

Escalate

Controls who can push a case to the next level of contacts when action is needed.

Resend Escalation / Follow-Up

Controls who can drive additional outreach without changing the escalation level.

History

Controls who can see the full audit trail for a case.

Read-only confusion

If a resource is enabled but not fully configured, it becomes Read Only. That is expected behavior, not an error. This trips people up constantly.

3 common configurations

These cover most organizations. Start with one of these before you start layering in special cases.

Scenario 1

Detectives can only update cases

  • Enable: Cases
  • Allow: Update
  • Disable: Delete, Archive, Escalate
Scenario 2

Lab Admin manages workflow

  • Enable: Cases
  • Allow: Assign, Escalate, Send Follow-Up
  • Optional: Create, Update
Scenario 3

Manager = visibility only

  • Enable: Dashboard and Reports
  • Set most things to: Read Only
  • Avoid: Operational case permissions unless necessary

Final take

Keep it simple: Admin = control, Lab Admin = workflow, Detective = execution, Manager = visibility. Everything else is configuration detail.
